## 2026-06-07 — iter-1: `example` Qlib backtest subcommand

- Added `zcrypto example`: a self-contained, **offline** Qlib strategy experiment on a tiny bundled crypto dataset — the project's first real Qlib wiring (`qlib.init` was previously absent).
- `cli/example/dataset.py` `build_provider` writes Qlib's file format (`calendars/day.txt`, `instruments/all.txt`, `features/<coin>/<field>.day.bin` as little-endian float32 with a start-index header) from an OHLCV CSV/`.csv.gz`. Each instrument's bin spans the **full calendar** (absent dates written as NaN) so multi-symbol datasets with differing date sets can never silently misalign.
- `cli/example/workflow.py` `run_experiment` runs the pipeline: ~6-feature `QlibDataLoader` handler + numpy-only `LinearModel` (OLS) → `SignalRecord`/`PortAnaRecord` backtest with `TopkDropoutStrategy(topk=2, n_drop=1)` over 6 coins (`REG_US`, `deal_price=close`, costs 0.0005/0.0015, benchmark `ETHUSD`); returns annualized-return / information-ratio / max-drawdown for strategy-absolute and excess-vs-ETH (with/without cost). Sets `MLFLOW_ALLOW_FILE_STORE=true` (MLflow ≥2 file store) and uses backtest `end_time = TEST end − 1 day` (Qlib needs a trading day after the last order to execute).
- `cli/example/command.py` runs everything inside a `TemporaryDirectory` (the Qlib binary data **and** the MLflow recorder are ephemeral; nothing persists in the repo). Heavy imports are deferred so `zcrypto --version` stays fast. `--show-data/--no-show-data` prints the prepared feature-frame head.
- Bundled data: `cli/example/data/crypto_ohlcv.csv.gz` — **gzip-compressed** (~40 KB) daily OHLCV, 6 coins (BTC/ETH/BNB/SOL/XRP/ADA), window `2025-12-01`..`2026-05-31` (182 dates each), generated one-off by dev-only `cli/example/scripts/gen_example_data.py` (`yfinance`; not a project dependency, not shipped). `build_provider` reads the gzip transparently via pandas; hatchling ships the file in the wheel.
- `README.md`: added a `## Requirements` section (OpenMP runtime — `brew install libomp` / `apt-get install libgomp1`, required because Qlib imports LightGBM) and documented the `example` command under `## Usage`.
- Tests use seeded synthetic data (no network): `tests/test_example_dataset.py` (binary round-trip + staggered multi-symbol alignment), `tests/test_example_workflow.py` (finite-metrics integration), `tests/test_example_command.py` (CLI smoke). Full suite 10 passed, 98% coverage.

## 2026-06-08 — iter-2: general CLI logger

- Added a project-wide CLI logger in `cli/logging/`: `configure(path, level)` attaches a single handler to both `logging.getLogger("zcrypto")` and `logging.getLogger("qlib")` (`propagate=False`), and calls `qlib.log.set_global_logger_level(numeric)` so qlib's internal manager respects the same threshold. Idempotent across repeated calls.
- Format follows destination: with `-l/--log <path>`, JSONL is appended to `<path>`; without `-l`, plain-text lines go to stdout. Plain-text format: `%(asctime)s %(levelname)s %(name)s [%(filename)s:%(lineno)d] - %(message)s` (qlib-style minus PID/thread). JSONL fields: `ts` (UTC ISO-8601, ms, trailing `Z`), `level`, `logger`, `file`, `line`, `message`, optional `extra` and `exception` (and `default=str` so `Path`/numpy scalars never crash logging).
- Added two global flags on the root callback in `cli/__main__.py`: `-l/--log <path>` and `--log-level {DEBUG,INFO,WARNING,ERROR}` (default `INFO`, case-insensitive). `--log-level` is an eager callback so `--log-level TRACE --version` errors at parse time instead of being swallowed by `--version`'s eager exit. `--version` itself keeps its existing eager fast path.
- Qlib's `dictConfig` would otherwise clobber the handler `configure()` installs — `cli/example/workflow.py`'s `qlib.init(...)` now passes `logging_config=None` to suppress qlib's reset so our handlers survive across `qlib.init()`.
- Migrated `cli/example/workflow.py`: the `--show-data` debug `print(...)` is now `logger.info("show_data: feature head", extra={"head": dataset.prepare("train").head().to_dict()})`. `_render`'s metrics table stays on `typer.echo` (command result, not a log).
- README `## Usage` documents the new flags. Tests: `tests/test_logging_formatters.py` (formatter shapes incl. underscore-key exclusion from `extra`), `tests/test_logging_get_logger.py` (namespace gate), `tests/test_logging_config.py` (wiring + idempotency + qlib clamp + invalid-level guard + end-to-end JSONL write), `tests/test_logging_cli.py` (CLI integration: --version stays fast; -l writes JSONL with qlib lines captured; no -l writes plain text on `result.stdout`; bad --log-level errors). Full suite 29 passed, 98% coverage.
- After merging develop (iter-3 + the MLflow `chdir` hot-fix), `configure()` also installs a `warnings.filterwarnings("ignore", "Mean of empty slice", RuntimeWarning, "qlib.utils.index_data")` to silence a benign qlib `np.nanmean` warning; `docs/open-topics/T0000-qlib-empty-slice-warnings.md` tracks the upstream issue so the filter can be removed once qlib guards the empty-slice case.

## 2026-06-09 — iter-4: `zcrypto data download` & `data verify`

- Added `cli/data/` subpackage (`config.py`, `klines.py`, `qlib_writer.py`, `index.py`, `snapshots.py`, `binance.py`, `verify.py`, `pipeline.py`, `command.py`) registered as a `data` Typer sub-app on the root app. Bare `zcrypto data` prints this group's help and exits (git-like one-level scoping); the full nested reference lives only in README `## Usage`.
- `zcrypto data download OUT_DIR PAIRS_FILE [--interval 1d] [--from 2020-01-01] [--to <yesterday UTC>]` fetches Binance spot 1d klines from `data.binance.vision`, sha256-validates each daily zip against its `.CHECKSUM`, and writes a Qlib-ready dataset directly into `OUT_DIR` — `calendars/day.txt`, `instruments/all.txt`, `features/<sym>/<field>.day.bin` for 11 fields (open, high, low, close, volume, amount, trades, taker_buy_base, taker_buy_amount, derived `vwap`, constant `factor=1.0`), plus `index.json`. Reconcile-to-file semantics: pairs in the file but not in the index get full history fetched (binary-search the listing date); existing pairs are time-extended (`--from ≤ index.to` → warn + adjust; `--from > index.to + 1` → gap error); indexed pairs absent from the file → error pointing to a future `delist`/`rename` command. A truncation guard rejects `--to < calendar.to` to prevent silent data loss. Symbol case is normalized to uppercase at parse time. `from_date > to_date` is rejected.
- `zcrypto data verify OUT_DIR [--silent]` re-validates an existing dataset against the index + every invariant and returns a structured `VerifyReport` from the pure `verify_dataset()` function — usable directly from any Qlib pipeline. `--silent` conveys the result via exit code only.
- Date arguments (`--from`, `--to`) are validated at parse time via a regex (`^\d{4}-\d{2}-\d{2}$`) + `dt.date.fromisoformat()`, so `20240101` or `2024-13-40` is rejected before any I/O.
- Every mutating subcommand (currently `download`) packs the whole relevant fileset (`calendars/`, `instruments/`, `features/`, `index.json`) into a single `OUT_DIR/.snapshots/<UTCstamp>-<cmd>.tar.gz` atomically (write-to-tmp + `os.replace`), retaining the newest 7. Staging happens in `OUT_DIR/.staging/`; the final `index.json` write also goes through the atomic tmp+rename path.
- **Commit-phase crash recovery.** The commit phase wraps file moves + index swap in a `try/except` that, on any in-process failure, restores the pre-commit state from the snapshot just taken — the operator sees the previous dataset, never a half-committed one. For out-of-process crashes (SIGKILL, power-off), a `OUT_DIR/.commit-in-progress` marker (atomic tmp+`os.replace` write) names the snapshot to restore from; the next `download_pipeline` run detects the marker and auto-recovers before any new work. `verify_dataset` also flags a leftover marker as a problem.
- `cli/data/binance.py` exposes a `Source` Protocol behind a stdlib `urllib.request` `BinanceSource` (HTTP paths excluded from coverage via `# pragma: no cover`); tests inject a `FakeSource` from `tests/data_fixtures.py` so the suite is **fully offline**.
- Daily zips are fetched via a bounded `ThreadPoolExecutor` (default **5** workers, set by `CliConstants.FETCH_CONCURRENCY` in `cli/constants.py`). The pool runs every `(pair, date)` tuple in parallel; per-pair gap-checking runs after the per-pair rows are collected and sorted. First failure cancels queued futures and propagates as `PipelineError`. This brings a 5-year × 20-pair backfill from ~36,000 sequential GETs (tens of minutes) to roughly 5-way parallel (single-digit minutes), without hammering `data.binance.vision`. Monthly-archive bulk fetch is still deferred as a future optimization.
- New `cli/constants.py` introduces a `CliConstants` class to host low-/no-change operational tuning (currently `FETCH_CONCURRENCY`). The repo convention: high-change-odds config gets a Typer flag; low-change-odds config lives in `CliConstants` and is tuned by editing the file.
- The first commit of this iteration also added `docs/research/01.binance-eea-spot-quant.md` (the strategy roadmap motivating the data work) and registered it in the `mdformat` pre-commit `files:` regex so its TOC stays auto-maintained.
- Future iteration follow-ups (deferred, not in scope): `backfill`, `delist`, `rename` subcommands; concurrent / monthly-archive fetcher. Coverage and tests are green at iteration close.
- README `## Usage` documents the new commands. Full suite: **104 passed** (29 pre-existing + 75 iter-4 tests across the 10 new `tests/test_data_*.py` and `tests/test_constants.py` files, including 1 concurrency test + 1 constants test + 4 commit-phase crash-recovery tests). `uv run pre-commit run --all-files` is green.

## 2026-06-09 — iter-5: data backfill/delist/rename and status-aware download/backfill

- Mutation harness `_execute_mutation(out_dir, cmd_name, plan_fn, apply_fn, *, dry_run)` consolidates the iter-4 recover→pre-verify→snapshot→commit discipline. All four mutators (download/backfill/delist/rename) now go through one place; the no-op short-circuit means snapshots are written only when real mutation occurs. Iter-4's `download_pipeline` refactored to fit this shape — all 104 iter-4 tests continue to pass.
- `zcrypto data backfill OUT_DIR [--to YYYY-MM-DD] [--dry-run]` extends every pair in the index forward to `--to` (default yesterday UTC). Per-pair status-aware: `TRADING` pairs use iter-4's right-edge reachability check; non-`TRADING` (BREAK, etc.) pairs are silently skipped with an info log. No-op (all caught up or all skipped) → no snapshot.
- `zcrypto data delist OUT_DIR SYMBOL [--dry-run]` removes a pair with conditional calendar shrink (front-trim rewrites every remaining bin's `start_index` header; back-trim shortens calendar without touching headers). Refuses on not-in-index, last-pair, gap-creating.
- `zcrypto data rename OUT_DIR OLD_SYMBOL NEW_SYMBOL [--dry-run]` relabels a pair. Variant 1: OLD in index only — probe NEW's first archive day on `data.binance.vision`; synthesize the gap as suspension bars (OHLC/VWAP = NaN — qlib's not-tradable marker, volume/amount/trades = 0, factor = 1.0). Variant 2: both in index — merge OLD's bin into NEW's slot, with the same synthetic gap fill in between. Same command, detected from the index.
- `validate_pairs_against_exchange` now reads Binance `status` (TRADING / BREAK / HALT / etc.) and routes downstream behavior: `download` fetches non-`TRADING` pairs as historical archive only via the new `find_available_range` helper; `backfill` silently skips them.
- `--dry-run` added to all four mutators. Skips snapshot, marker, and staging; prints the plan to stdout via `typer.echo`. Errors out if `.commit-in-progress` marker is present.
- Snapshot tar.gz files named `<UTCstamp>-<cmd>.tar.gz` consistently for every mutator.
- `CliConstants.RENAME_SYNTH_WARN_DAYS = 7` controls when a rename gap warning is elevated.
- Test count: **162 passed** (104 iter-4 baseline + 58 iter-5 across `test_data_backfill.py`, `test_data_delist.py`, `test_data_rename.py`, `test_data_e2e.py`, and extensions to `test_data_pipeline.py` / `test_data_command.py` / `test_data_download.py`).

## 2026-06-18 — iter-9: validation rigor (CPCV)

- CPCV is now the **default** `experiment` run: purged k-fold CV with an embargo sized to `label_horizon_days` + `feature_lookback_days` is used to generate combinatorial backtest paths, replacing the prior single train→valid→test split as the primary validation method.
- New `cli/experiment/cv.py`: pure split engine (no qlib dependency) — groups the calendar into N equal blocks, enumerates all C(N, k) test-group combinations, applies purge/embargo around each test block, and stitches per-split predictions into C(N-1, k-1) full-span backtest paths via `build_cv_plan` / `assemble_paths`.
- New `cli/experiment/cpcv.py`: qlib orchestration layer — materializes the Alpha158 feature/label matrix once over `train+valid`, trains a LightGBM booster per purged+embargoed split (fixed rounds, no early stopping inside CV folds), predicts held-out rows, assembles paths, runs signal-driven `backtest()` per path, and aggregates a Sharpe distribution (+ rank-IC) into a `CPCVResult`.
- Four new `Recipe` CV fields added to `cli/experiment/recipes/base.py`: `label_horizon_days` (default 2), `feature_lookback_days` (default 60), `cv_n_groups` (default 6), `cv_test_groups` (default 2).
- `--quick` flag on `zcrypto experiment`: skips CPCV and runs only the single holdout backtest on `test` (today's prior behavior).
- New output artifacts: `cv_results.json` (per-path Sharpe / return / max-DD distribution + rank-IC stats + holdout summary), a 4th report panel (histogram of path Sharpes with a vertical line at the holdout Sharpe), and a `CPCV (<N> paths, train+valid): Sharpe <mean> ± <std> (worst <worst>) · rank-IC <mean>` stdout line.
- New JSONL log events emitted during a CPCV run: `cpcv-init`, `cpcv-start`, `split-trained`, `paths-assembled`, `path-backtest`, `cv-aggregated`, `cpcv-done`.
- `scaffold._redis_preflight` renamed to public `redis_preflight` and moved out of the private namespace; it is now shared by both the holdout scaffold and `cpcv.py`.
- The open-topics convention (`.claude/rules/open-topics.md`) gained a `partial` status as a third lifecycle state alongside `open` and `resolved`; topic `T0002` (validation rigor) was transitioned to `partial` — purged k-fold + embargo and CPCV landed in this iteration, while deflated Sharpe + PBO, the multi-recipe ranking surface, and MLFinLab remain deferred.

## 2026-06-18 — iter-11: deflated Sharpe, PBO, and the rank command

- New `cli/experiment/stats.py` (pure, no qlib): `sharpe`, `psr` (Probabilistic Sharpe Ratio — P(true SR > benchmark), corrected for sample length + skewness + kurtosis), `expected_max_sharpe` (Euler-gamma approximation), `deflated_sharpe` (PSR with N-trials correction), and `pbo_cscv` (Probability of Backtest Overfitting via Combinatorially-Symmetric CV). All functions operate on plain numpy arrays; scipy.stats (already present via pyqlib) provides `norm.cdf`/`norm.ppf`.
- Per-run **holdout PSR** surfaced in three places: `cv_results.json` (`holdout.psr`), the report's holdout vertical-line annotation (appended after "test period"), and a `holdout PSR` stdout line. Every run also writes `returns.csv` (`date,ret` — holdout cost-adjusted daily returns) for downstream consumption by `zcrypto rank`.
- New top-level `zcrypto rank` command (`cli/experiment/rank.py`): scans every `<out>/<recipe>/<run>/returns.csv` as a trial, aligns trials on their common date window, and reports the **deflated Sharpe ratio** of the best trial + **PBO** (CSCV, default 16 splits) with a ranked table. Writes `runs/rank.json` with keys `n_trials`, `window`, `n_splits`, `trials` (per-trial `recipe`/`run`/`sharpe`/`psr`), `dsr_best`, `pbo`. DSR and PBO are `NaN` for fewer than 2 trials. Log events: `rank-scan`, `rank-aligned`, `rank-done`.
- Report relabels: the 4th-panel title changed from "CPCV out-of-sample Sharpe distribution" to "CPCV OOS Sharpe distribution (descriptive)" — making clear the band is not a confidence interval; the holdout vertical-line annotation changed from "holdout" to "holdout (test period) · PSR <value>" — making clear it marks a different-period reference rather than an overfitting test.
- open-topic `T0002` (validation rigor) resolved: the two iter-9 CPCV interpretation caveats are retired — the CPCV path-Sharpe band is now labelled descriptive and the holdout marker is relabelled a different-period reference; DSR/PBO are the honest overfitting measures.
